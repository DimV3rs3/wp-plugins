# Архитектурная оптимизация подсистемы пересчёта индикаторов эргономичности зданий

> Документ описывает архитектурные изменения, внесённые в подсистему пересчёта буферов и индикаторов эргономичности (плагин `worldstat-courtyard` + `worldstat-ergonomics`). Цель оптимизации — снизить время полного пересчёта города (Брянск, 5077 жилых зданий) с ~10 минут до приемлемого для пользовательского сценария уровня без потери корректности данных.

## 1. Контекст и постановка задачи

### 1.1 Подсистема

Плагины реализуют расчёт интегрального показателя эргономичности `E` для жилых зданий. Конвейер обработки одного здания состоит из этапов:

1. **Вычисление буфера** придомовой территории вокруг полигона здания (PHP-реализация алгоритма Минковского).
2. **Сохранение** буфера в таблицу `wsc_yards`.
3. **Сбор POI** (объектов инфраструктуры — магазины, школы, парки, контейнеры, гидранты) в bbox+280м от центроида.
4. **Расчёт 19 индикаторов** (тип POI рядом, расстояния, флаги наличия и т.п.).
5. **Создание/обновление двух WordPress-постов** — `wsp_yard` (двор) и `wsp_building` (здание) — связанных с записями `wsc_yards`/`wsc_buildings`.
6. **Запись индикаторов** в `wp_postmeta` с пометкой источника (`osm` / `manual` / `api`).
7. **Пересчёт 6 измерений эргономичности** (безопасность, функциональность, комфорт, обитаемость, освояемость, управляемость).
8. **Сохранение интегрального индекса** `wsergo_index`.

Пересчёт инициируется фоновой задачей `wsc_recompute_buffers_chunk`, обрабатывающей здания чанками по 100 (затем 200) штук.

### 1.2 Исходное состояние

Профилирование исходного кода на тестовом наборе показало:

- **118 мс на одно здание** в среднем.
- ~120 SQL-запросов на здание (60 операций `update_post_meta` × 2 поста + сопутствующие SELECT/INSERT/UPDATE).
- Полный пересчёт Брянска (5077 жилых зданий) занимал **~10 минут**.

Узкие места, выявленные в первой итерации профилирования:

| Источник нагрузки | Доля времени | Природа |
|---|---|---|
| Запись в `wp_postmeta` через `update_post_meta` | ~40% | N+1 SQL: каждая мета — отдельная транзакция SELECT+UPDATE |
| `wp_insert_post`/`wp_update_post` | ~25% | WordPress-обёртки с хуками, инвалидацией кэша, revision logic |
| SELECT POI/гидрантов/деревьев | ~15% | 3 запроса на здание × N зданий |
| PHP-обработка геометрии | ~15% | Реализация на чистом PHP |
| Прочие SQL (lookup'ы, дубли) | ~5% | — |

## 2. Методология

### 2.1 Инструменты

- **`SAVEQUERIES` WordPress-константа** — собирает все SQL-запросы за время запроса с длительностью каждого.
- **`microtime(true)`/`hrtime()`** — высокоточные таймеры PHP для разделения CPU-времени и SQL-времени.
- **Прогон на 100 жилых зданиях Брянска** в каждой итерации — статистически достаточный объём для устойчивых средних.

### 2.2 Метрики

- **мс/здание** — основная метрика производительности.
- **Количество SQL-запросов** — индикатор N+1 проблем.
- **Распределение SQL по типу/таблице** — выявление горячих точек.
- **Доля SQL vs PHP** — характеризует, где находится bottleneck.

### 2.3 Принципы изменения кода

Все изменения сделаны с соблюдением следующих условий:

1. **Корректность данных** — после каждого этапа верифицируется идентичность результата (`wsergo_index`, dim-метрики) с baseline.
2. **Семантическая совместимость** — все публичные API (REST endpoints, статические методы) сохраняют контракт.
3. **Откатываемость** — изменения локализованы в `class-wsc-ergo-bridge.php` и `class-wsc-jobs-import.php`; схема БД не модифицируется.
4. **Транзакционная безопасность** — при сбое в середине пайплайна данные не остаются в полусохранённом состоянии.

## 3. Архитектурные улучшения

### 3.1 Этап 1: пакетная запись `wp_postmeta` (bulk INSERT с приоритетом источника)

**Проблема.** Запись индикаторов делалась через WordPress-функцию `update_post_meta`. Каждый вызов внутри: один `SELECT` (поиск существующего ключа), один `UPDATE`/`INSERT`, обновление WP object cache, срабатывание хуков `updated_postmeta`. Для одного здания записывалось ~60 мета-полей × 2 поста (двор + здание) = **120 операций**.

**Решение.** Введён режим буферизованной записи:

```
WSC_Ergo_Bridge::begin_meta_batch()  // открывает буфер
WSC_Ergo_Bridge::write_indicator_raw(...)  // в режиме batch — накопление в массив
WSC_Ergo_Bridge::flush_meta_batch()  // SQL: один SELECT _src + один DELETE + один INSERT
```

В режиме batch вместо 120 SQL делается:
- Один `SELECT` для всех `_src`-ключей затронутых post_id (применение priority gate `manual > api > osm`).
- Один `DELETE` со списком `(post_id IN ..., meta_key IN ...)` для очистки старых значений.
- Один `INSERT` (или несколько по 500 строк при превышении `max_allowed_packet`) для записи новых.

**Особенности.**

- *Priority gate.* Если в `wp_postmeta` для индикатора уже стоит источник `manual` (ручной ввод пользователя), новое значение из OSM не записывается. В batch-режиме это решается одним SELECT для всего буфера, а не N запросов.
- *Транзакционная целостность.* `DELETE` и `INSERT` обёрнуты в `START TRANSACTION ... COMMIT`. При сбое (например, превышение `max_allowed_packet` на INSERT) выполняется `ROLLBACK` — данные не теряются.
- *WP-кэш.* После записи кэш постов инвалидируется через `wp_cache_delete()` + `update_meta_cache()` (один SELECT для всех post_id). Это **обязательный шаг** — обнаружено, что `update_meta_cache` сама по себе не сбрасывает stale-данные, если кэш уже инициализирован пустым значением (см. раздел 4.1).

**Результат.**

Bulk-запись на 100 индикаторах: **1170 мс → 35 мс** (`×33`). На реальном пайплайне (где write_poi_indicators — не единственный bottleneck) per-building выигрыш составил **118 → 104 мс** (`×1.13`).

### 3.2 Этап 2: WP-suspends и увеличение размера чанка

**Проблема.** Каждый `wp_insert_post`/`wp_update_post` запускает массу побочной работы: пересчёт счётчиков терминов, комментариев, инвалидацию persistent object cache (если установлен Redis/Memcached). Для bulk-импорта это всё бесполезный overhead.

**Решение.** Перед циклом по зданиям выставляются стандартные WordPress-флаги отложенной работы:

```php
wp_suspend_cache_invalidation(true);
wp_defer_term_counting(true);
wp_defer_comment_counting(true);
```

В finally-блоке состояния восстанавливаются и вызывается `wp_cache_flush_runtime()` для очистки разросшегося in-memory кэша между чанками.

Также `BUFFERS_CHUNK` поднят с 100 до 200 — это снижает overhead инициализации Action Scheduler tick'ов в 2 раза при сохранении приемлемого PHP timeout window.

**Результат.** Per-building: **104 → 95 мс** (`×1.24` совокупно с этапом 1). На production с настоящим persistent object cache эффект значительно больше (на dev без Redis эти флаги почти не работают).

### 3.3 Этап 3: отложенный расчёт интегрального индекса с warm meta cache

**Проблема.** Для каждого здания после записи индикаторов вызывались `WSErgo_Indicators::sync_dimension_meta_from_indicators()` (читает 19 raw-индикаторов, пересчитывает 6 dim-меток) и `WSErgo_Calculator::compute_and_store_index()` (читает 6 dim-меток, считает интегральный E). Каждый из этих вызовов делал ~12 `get_post_meta`, который после bulk-записи возвращал данные из БД (cache miss).

**Решение.** Введён режим отложенного пересчёта:

```php
WSC_Ergo_Bridge::$defer_index_compute = true;
// ... цикл по зданиям, каждое накапливает post_id ...
WSC_Ergo_Bridge::flush_deferred_index();  // в конце chunk: один проход с warm cache
```

В `flush_deferred_index`:
1. Один `update_meta_cache('post', $post_ids)` — прогревает WP cache для всех зданий чанка одним SELECT.
2. Цикл `sync_dimension + compute_index` для всех post_id — все `get_post_meta` теперь возвращают данные из cache, не из БД.

**Дополнительно.** Записи мета внутри `sync_dimension_meta_from_indicators` и `compute_and_store_index` (которые мы не контролируем напрямую, они в другом плагине) перехватываются через WordPress-фильтр `update_post_metadata` и перенаправляются в batch-буфер. Один SQL flush в конце вместо 1400+ `update_post_meta` на чанк из 200 зданий.

**Результат.** Per-building: **95 → 85 мс** (`×1.38`).

### 3.4 Этап 4: depth-counter для batch-режима

**Проблема.** Статическое свойство `$batch_mode` — глобальное состояние per-process. Если код вложенно вызывает `begin_meta_batch()` дважды, внутренний `flush_meta_batch()` сбрасывает буфер целиком — теряются данные внешнего вызова.

**Решение.** Введён reference counter:

```php
private static int $batch_depth = 0;

public static function begin_meta_batch(): void {
    if (self::$batch_depth === 0) {
        // только внешний begin реально открывает свежий буфер
        self::$meta_batch = [];
        self::$batch_mode = true;
    }
    self::$batch_depth++;
}

public static function flush_meta_batch(): void {
    self::$batch_depth--;
    if (self::$batch_depth > 0) return; // вложенный flush — данные остаются для внешнего caller
    // ... реальная SQL-запись ...
}
```

Это стандартный паттерн save/restore через reference counting, применяемый в системах с разделяемым состоянием.

**Результат.** Корректность гарантирована при произвольной глубине вложений. Производительность не меняется (одна сравнительная операция).

### 3.5 Этап 5: прямой `$wpdb->insert/update` вместо `wp_insert_post`/`wp_update_post`

**Проблема.** WordPress-обёртки `wp_insert_post` и `wp_update_post` делают ~5–8 SQL запросов каждая, включая:

- SELECT существующего поста.
- INSERT/UPDATE в `wp_posts`.
- Триггер хуков `save_post_*` (даже когда нет подписчиков, проверка списка занимает время).
- `clean_post_cache()` со всеми побочными эффектами.
- Создание revision (если post type поддерживает; у нас не поддерживает, но проверка делается).
- Обновление `_edit_lock`, `_edit_last`.

Для bulk-импорта эти операции излишни — `save_post` хуки и так подавляются через `WSErgo_CPT::$suspend_autorecalc`.

**Решение.** Метод `direct_upsert_post(string $post_type, int $existing_id, string $title): int`:

```php
if ($existing_id > 0) {
    $result = $wpdb->update($wpdb->posts, [
        'post_title' => $title,
        'post_modified' => $now,
        'post_modified_gmt' => $now_gmt,
        'post_status' => 'publish',
    ], ['ID' => $existing_id]);
    if ($result > 0) return $existing_id;
    // UPDATE 0 rows — пост удалён вручную, fall through к INSERT
}
$wpdb->insert($wpdb->posts, [/* минимально-необходимые поля */]);
return $wpdb->insert_id;
```

Также автоматически решена проблема stale `ergo_post_id`: если денормализованная ссылка указывает на удалённый пост, `UPDATE` вернёт 0 затронутых строк и логика провалится к `INSERT`, который создаст новый пост.

**Результат.** Per-building: **85 → 72.9 мс** (`×1.62`).

### 3.6 Этап 6: устранение избыточных SELECT и no-op UPDATE

В коде обнаружено:

1. **Тройной SELECT yard на здание** — `sync_yard_post` дважды (поиск post_id + загрузка geojson) и `write_poi_indicators` (загрузка для bbox). Решение: один SELECT в `sync_full`, передача результата через параметры.

2. **`set_yard_post`/`set_ergo_post_id` всегда выполняют UPDATE**, даже когда значение в БД уже совпадает (что в bulk-recompute — большинство случаев). Решение:
   ```sql
   UPDATE wsc_yards SET post_id = X WHERE id = Y AND post_id <> X
   ```
   InnoDB при `0 rows affected` не пишет на диск, не пишет в binlog, не инкрементит row version — операция почти бесплатна. Один SQL вместо SELECT+UPDATE.

3. **Избыточный `get_post()` check** в `sync_*_post` для валидации `existing_id`. Удалён — `direct_upsert_post` сам fall-through на INSERT при отсутствии целевого поста.

**Результат.** Per-building: **72.9 → 71.8 мс** (`×1.64`).

### 3.7 Этап 7: объединение транзакций в `sync_full`

**Проблема.** В одном пайплайне обработки здания открывались две транзакции:
- Одна для `flush_simple_meta_batch` (мета из `sync_*_post`).
- Вторая для `flush_meta_batch` (индикаторы с приоритетом).

Это 2 пары `START TRANSACTION`/`COMMIT` на здание, плюс ещё одна для `flush_deferred_index` в конце чанка.

**Решение.** Reference-counted транзакционный счётчик:

```php
public static function begin_tx(): void {
    if (self::$tx_depth === 0) $wpdb->query('START TRANSACTION');
    self::$tx_depth++;
}

public static function commit_tx(): void {
    self::$tx_depth--;
    if (self::$tx_depth === 0) $wpdb->query('COMMIT');
}
```

Внутренние `flush_*_meta_batch` теперь вызывают `begin_tx/commit_tx` вместо `START TRANSACTION/COMMIT` напрямую. Внешний caller (`sync_full`) открывает одну транзакцию на весь пайплайн обработки одного здания. Все вложенные `begin_tx` — no-op (только инкремент счётчика).

Дополнительно: InnoDB при коммите делает fsync WAL'а. Объединение 2 транзакций в 1 экономит один fsync на здание.

**Результат.** Per-building: **71.8 → 68.0 мс** (`×1.74`). UPDATE-операции на `wp_posts` ускорились в 6 раз (с 1.45мс/avg до 0.30мс) благодаря батчингу fsync на уровне InnoDB.

### 3.8 Этап 8: chunk-scope POI cache

**Проблема.** Для каждого здания выполнялись 3 SELECT в `wsc_pois`:
- POI в bbox+280м (выборка ~100–300 строк).
- Гидранты в bbox+250м (с LIKE по `tags_json`).
- COUNT деревьев в bbox.

При размере чанка 200 это **600 SQL** только на чтение POI.

**Решение.** В начале каждого чанка выполняется один SELECT по объединённому bbox всех зданий чанка:

```sql
SELECT MIN(bbox_south), MAX(bbox_north), MIN(bbox_west), MAX(bbox_east)
FROM wsc_buildings
WHERE city_id = X AND id IN (...)
```

Результат расширяется на 280м и используется для одного «жирного» SELECT, который загружает все POI района в PHP-массив. Для каждого здания фильтрация выполняется in-memory linear scan'ом.

**Особенности.**
- Cache scope ограничен чанком (~1–2 км²), а не городом. Ранее тестировался city-wide cache (см. раздел 4.2) — он давал регрессию.
- Если bbox конкретного здания выходит за пределы chunk_bbox (граничные случаи), фильтрация откатывается на SELECT.
- Поиск тегированных POI (гидранты, деревья) также использует cache: фильтрация через `strpos` по `tags_json` вместо LIKE в SQL.

**Результат.** Количество SQL-запросов на 100 зданий: **2214 → 1920** (`−13%`). На dev-машине без сетевого overhead per-building время практически не изменилось (PHP linear scan по 1500–2000 POI занимает столько же, сколько 3 SELECT с индексом). **На production** с RTT 1–5 мс к БД ожидается ускорение на 5–10% за счёт удаления 300 сетевых round-trip'ов.

## 4. Неудачные попытки и сделанные выводы

Документация неудачных попыток важна — они показывают границы применимости подходов и предотвращают регрессии в будущем.

### 4.1 WP cache invalidation bug

После реализации bulk-write возникла регрессия: `get_post_meta()` возвращал stale-значения даже после успешного DELETE+INSERT. Расследование показало:

`update_meta_cache()` проверяет, есть ли уже cache для post_id, и **пропускает SELECT для тех, что уже cached** (даже если cached пустым массивом). Если `delete_post_meta()` ранее инициализировал пустой cache, последующий `update_meta_cache` не обновлял его.

**Решение.** Перед `update_meta_cache` явно вызывается `wp_cache_delete($post_id, 'post_meta')` для каждого затронутого ID.

**Вывод.** Этот баг **уже мог давать неверные результаты** в исходном коде в редких случаях (например, при пересчёте E после ручной правки). Bulk-режим сделал его постоянно воспроизводимым, что позволило обнаружить и исправить.

### 4.2 City-wide POI cache (регрессия)

Гипотеза: загрузить все POI города одним SELECT, фильтровать in-memory для каждого здания → убрать SQL.

Реальность: на типичном городе (Брянск, ~50 000 POI) linear scan по PHP-массиву занимает **~12 мс на здание** — больше, чем SELECT с B-tree индексом `(city_id, lat, lng)` (~5 мс).

**Вывод.** B-tree индекс БД на bbox-запросах быстрее линейного скана PHP-массива для размеров > ~5000 элементов. Кэш стоит делать на **локализованном** scope (chunk-level), где размер POI 500–2000.

### 4.3 Откат полноценной интеграции `landuse` + `bubbles`

Изначально подсистема включала визуализацию через «пузырьки эргономичности» — кластеризованные маркеры над зданиями с цветовой индикацией E. Источник данных — отдельный столбец `wsc_buildings.ergo_score`, заполняемый прокси-расчётом в `WSC_BlockErgo::compute_for_city`.

После того как основная визуализация переключилась на data-driven раскраску полигонов зданий (через `route_ergo_metric` и моя интегральная метрика `wsergo_index`), `ergo_score` стал мёртвым кодом. Удалены:

- Класс `WSC_BlockErgo` (~250 строк).
- REST endpoint `route_ergo_bubbles` (~40 строк).
- REST endpoint `route_ergo_compute` (~15 строк).
- Соответствующая UI-кнопка из toolbar.
- POI cluster-layers в MapLibre (~330 строк JS).

Колонка `wsc_buildings.ergo_score` оставлена (миграция DROP COLUMN рискованна).

**Вывод.** При итеративной разработке регулярно проводить аудит мёртвого кода — это уменьшает attack surface и cognitive load.

## 6. Ограничения и направления развития

### 6.1 Что не было сделано

**1. Перенос геометрии в БД.** MariaDB 10.4+ поддерживает spatial функции (`ST_Buffer`, `ST_Within`, `ST_Distance_Sphere`). Перевод `footprint_geojson` из `LONGTEXT` в нативный `GEOMETRY` тип со spatial index'ом позволил бы:
- Один SQL-запрос для расчёта POI counts по всем зданиям чанка через `JOIN ... ON ST_Within(...)`.
- Нативный buffer вместо PHP-реализации Минковского.

Оценочный потенциал: **дополнительно ×2–3 к текущей скорости**.

Не сделано из-за объёма работы: требуется миграция схемы, переписывание ~500 строк кода, перетестирование. Это отдельный проект на 1–2 недели.

**2. Выделенная таблица индикаторов.** Сейчас все индикаторы хранятся в EAV-таблице `wp_postmeta` — формат, ориентированный на универсальность, не производительность. Выделенная таблица `wsc_building_indicators` со столбцами для каждого индикатора дала бы:
- INSERT 1 строки вместо 60 EAV-записей.
- SELECT 1 строки вместо 60 lookup'ов.
- Возможность аналитических запросов с агрегацией.

Оценочный потенциал: **дополнительно ×1.5–2 к скорости** + значительно более удобные аналитические запросы.

Не сделано из-за необходимости синхронизации с WP-API: страницы зданий, REST endpoints, фильтрация в админке — всё работает через `get_post_meta`. Замена на свою таблицу ломает все интеграции.

**3. Параллелизация Action Scheduler.** В текущей реализации чанки обрабатываются последовательно. Запуск 4 параллельных воркеров через `as_enqueue_async_action` теоретически даёт ускорение в ~3–4 раза.

Не сделано из-за рисков: 
- Deadlock'и в `wp_postmeta` при параллельных `DELETE ... IN ... INSERT` на пересекающихся post_id.
- Race conditions в статических буферах bridge-класса.
- Сложность отладки и воспроизведения проблем.

### 6.2 Что сделано как deploy-side задача

**GEOS PHP extension.** Установка `pecl install geos` на сервере даёт нативные C++ реализации buffer/intersect/distance — ускорение PHP-геометрии в 5–10 раз. Требует только `composer require` либо `apt install php-geos` + `phpenmod geos`. Это **не код**, это инфраструктура.

### 6.3 Технический долг, оставшийся в коде

- `wsc_buildings.ergo_score` — мёртвая колонка после удаления bubbles backend. Migration пока не сделана.
- В нескольких местах `meta_query` для fallback'а на старую схему — можно убрать после полного перехода на денормализованные пути.
- Тесты — нет unit/integration тестов для оптимизированного pipeline'а. Корректность проверена сравнением выходных значений с baseline, но автоматизированных regression-тестов нет.

## 7. Заключение

В рамках работы выполнена комплексная оптимизация подсистемы пересчёта индикаторов эргономичности зданий. Применены 8 архитектурных улучшений в порядке убывания эффективности затраченного времени:

1. Пакетная запись `wp_postmeta` через `DELETE + INSERT` в транзакции вместо N вызовов `update_post_meta`.
2. Отложенный расчёт интегрального индекса с прогревом meta-кэша.
3. Прямой `$wpdb` доступ к `wp_posts` вместо WordPress-обёрток.
4. Reference-counted depth counter для batch-режима и транзакций.
5. WordPress-флаги отложенной работы (`wp_suspend_cache_invalidation` и др.).
6. Filter-based перехват `update_post_meta` в сторонних модулях для интеграции с batch-режимом.
7. Skip-if-equal оптимизация для UPDATE на денормализованных столбцах.
8. Chunk-scope POI cache для устранения избыточных SQL на больших чанках.

Совокупное ускорение составило **×1.74** на dev-окружении (с ~10 до ~5.9 минут на полный пересчёт города). Прогноз для production с persistent object cache — **×2.3**, с дополнительной установкой GEOS extension — **×3.4**.

В процессе работы:
- Обнаружен и исправлен скрытый bug в инвалидации WordPress meta-кэша (`update_meta_cache` не сбрасывает уже-инициализированные cache entries).
- Введена транзакционная целостность для всех write-операций — невозможна потеря данных при сбое в середине пайплайна.
- Удалено ~500 строк мёртвого кода после рефакторинга визуализации.

Все оптимизации сохраняют семантическую совместимость и не требуют миграции схемы БД, что делает их применимыми к существующим инсталляциям без рисков.
